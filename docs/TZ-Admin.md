# ТЗ: Администратор системы

**Модуль:** управление стандартами, моделью данных и связями
**Версия:** 1.0
**Статус:** На согласовании

---

## 1. Цель

Обеспечить возможность конфигурирования системы ESG-отчетности, включая:

- полное представление стандартов раскрытия;
- декомпозицию требований до атомарного уровня;
- настройку сквозных (shared) элементов данных;
- настройку связей между требованиями и данными;
- настройку логики объединения (merge) стандартов;
- настройку дополнительных требований (дельт);
- определение правил полноты и валидации.

---

## 2. Область применения

Функциональность применяется для:

- подготовки системы к использованию конечными пользователями;
- поддержки мультистандартной отчетности (например, GRI + IFRS S2 + SASB);
- обеспечения методологической целостности и непротиворечивости данных;
- централизованного управления требованиями и логикой раскрытий.

---

## 3. Функциональные требования

### 3.1. Управление стандартами

Система должна позволять:

- создавать и редактировать стандарты;
- хранить версии стандартов;
- задавать структуру стандарта (разделы, подразделы);
- импортировать структуру стандарта;
- просматривать структуру в виде дерева;
- деактивировать устаревшие версии.

**Связь с БД:** `standards`, `standard_sections`

**UI:** `/settings/standards` — список стандартов, дерево секций

### 3.2. Управление disclosure-требованиями

Система должна позволять:

- создавать disclosure requirements;
- задавать:
  - код (GRI 305-1, IFRS S2.29);
  - название;
  - описание;
  - тип (`quantitative` / `qualitative` / `mixed`);
  - уровень обязательности (`mandatory` / `conditional` / `optional`);
  - условия применимости (`applicability_rule` — JSON);
- связывать требования с разделами стандарта;
- управлять порядком отображения.

**Связь с БД:** `disclosure_requirements`

### 3.3. Декомпозиция требований

Система должна позволять:

- разбивать disclosure на атомарные requirement items;
- задавать иерархию элементов (`parent_item_id`);
- задавать тип элемента:

| item_type | Описание | Пример |
|-----------|----------|--------|
| `metric` | Числовой показатель | Scope 1 total (t CO2e) |
| `attribute` | Атрибут / характеристика | Boundary approach |
| `dimension` | Разрез / breakdown | By gas type |
| `narrative` | Текстовое описание | Methodology description |
| `document` | Требуемый документ | Audit certificate |

- задавать:
  - тип значения (`number`, `text`, `boolean`, `date`, `enum`, `json`);
  - единицу измерения (`unit_code`);
  - обязательность (`is_required`);
  - кратность (`cardinality_min`, `cardinality_max`);
  - правила гранулярности (`granularity_rule` — JSON);
  - правила валидации (`validation_rule` — JSON).

**Связь с БД:** `requirement_items`

### 3.4. Управление shared data elements

Система должна позволять:

- создавать сквозные элементы данных;
- задавать:
  - код (`GHG_SCOPE_1_TOTAL`);
  - домен (`emissions`, `energy`, `water`, `waste`, `workforce`, `governance`);
  - тип значения;
  - единицу измерения;
  - описание;
- задавать допустимые измерения (dimensions):
  - scope, category, gas, geography, facility;
- определять обязательные измерения.

**Связь с БД:** `shared_elements`, `shared_element_dimensions`

### 3.5. Настройка связей (mapping)

Система должна позволять:

- связывать requirement items со shared elements;
- задавать тип связи:
  - `full` — полное покрытие;
  - `partial` — частичное (с дельтами);
  - `derived` — расчётное значение;
- просматривать:
  - какие требования используют один shared element;
  - какие shared elements участвуют в нескольких стандартах;
  - «карту пересечений» стандартов.

**Связь с БД:** `requirement_item_shared_elements`

**UI:** матрица маппинга (shared element × стандарт)

### 3.6. Настройка merge стандартов

Система должна позволять:

- выбирать набор стандартов для проекта;
- определять базовый стандарт (`is_base_standard`);
- выявлять пересечения требований (через shared elements);
- подтверждать совпадающие элементы;
- формировать unified requirement set;
- сохранять конфигурации объединения;
- отображать различия между стандартами.

**Связь с БД:** `reporting_project_standards`, `requirement_item_shared_elements`

**UI:** Merge View — матрица element × standard

### 3.7. Управление delta-требованиями

Система должна позволять:

- создавать дополнительные требования при комбинации стандартов;
- определять тип дельты:
  - `additional_item` — дополнительное поле;
  - `extra_dimension` — дополнительная гранулярность;
  - `stricter_validation` — усиленная обязательность;
  - `extra_narrative` — дополнительный narrative;
  - `extra_document` — дополнительный документ;
- задавать условия применения дельты;
- привязывать дельты к конкретным стандартам и requirement items.

**Связь с БД:** `requirement_deltas`, `requirement_item_overrides`

### 3.8. Настройка правил полноты

Система должна позволять:

- определять критерии статусов:
  - `complete` — все обязательные элементы заполнены и approved;
  - `partial` — есть данные, но неполные или не approved;
  - `missing` — данных нет;
  - `not_applicable` — не применимо (с обоснованием);
- задавать:
  - обязательные поля;
  - обязательные dimensions;
  - требования к narrative;
  - требования к evidence;
- задавать правила расчета completion percent.

**Связь с БД:** `requirement_item_statuses`, `disclosure_requirement_statuses`

### 3.9. Настройка зависимостей

Система должна позволять:

- задавать зависимости между requirement items:
  - `requires` — если A заполнен, требуется B;
  - `excludes` — A и B взаимоисключающие;
  - `conditional_on` — B обязателен при условии;
- задавать условные правила (`condition_expression` — JSON):
  - if/then;
  - conditional requirements;
- учитывать контекст:
  - выбранные стандарты;
  - наличие значений;
  - тип компании;
  - отрасль.

**Связь с БД:** `requirement_item_dependencies`

### 3.10. Настройка пользовательских форм

Система должна позволять:

- **автоматически генерировать** формы ввода из requirement_items;
- администратор конфигурирует (не пишет код):
  - группировку полей (по стандарту / по теме / по shared элементам);
  - порядок полей;
  - подсказки и описания;
  - видимость технических элементов;
  - обязательность отображения;
- скрывать технические элементы от сборщиков данных.

> **Принцип:** формы ввода = проекция requirement_items на UI. Администратор управляет представлением, не структурой.

### 3.11. Управление справочниками

Система должна позволять создавать и редактировать справочники:

| Справочник | Таблица | Примеры значений |
|-----------|---------|-----------------|
| Единицы измерения | (unit_codes) | t CO2e, MWh, m3, %, шт |
| Scope | (dimension values) | Scope 1, Scope 2, Scope 3 |
| Типы газов | (dimension values) | CO2, CH4, N2O, HFCs, PFCs, SF6 |
| Scope 3 категории | (dimension values) | Category 1–15 |
| Boundary types | `boundaries` | Operational control, Financial control, Equity share |
| Методологии | `methodologies` | GHG Protocol, Location-based, Market-based |

Справочники используются в requirement_items и data_points.

### 3.12. Управление правами доступа

Система должна позволять:

- назначать роли пользователям (`admin`, `esg_manager`, `collector`, `reviewer`, `auditor`);
- настраивать права по матрице (см. основное ТЗ, раздел 6.1);
- назначать метрики на сборщиков и ревьюеров (`metric_assignments`);
- массовое назначение через матрицу.

**Связь с БД:** `users`, `metric_assignments`

**UI:** `/settings/users`, `/settings/assignments`

## 4. Пользовательские сценарии (администратор)

### Сценарий 1. Заведение стандарта

1. Создание стандарта (code, name, version)
2. Загрузка / создание структуры секций
3. Создание disclosure requirements
4. Декомпозиция каждого disclosure на requirement items
5. Задание типов, единиц, правил валидации

**Результат:** стандарт полностью описан и готов к использованию

### Сценарий 2. Настройка shared элемента

1. Создание shared element (code, concept_domain, unit)
2. Определение допустимых dimensions
3. Привязка к requirement items из разных стандартов (mapping)
4. Проверка: маппинг корректен (full/partial)

**Результат:** элемент доступен для reuse между стандартами

### Сценарий 3. Настройка merge

1. Создание reporting project
2. Выбор стандартов (GRI + IFRS S2)
3. Система показывает пересечения (через shared elements)
4. Администратор подтверждает совпадения
5. Администратор добавляет дельты (IFRS requires financial impact)
6. Сохранение конфигурации

**Результат:** unified requirement set сформирован, дельты зафиксированы

### Сценарий 4. Изменение правила (impact analysis)

1. Администратор изменяет requirement item (например, делает optional → mandatory)
2. Система показывает: «Это повлияет на 3 проекта, 12 data points»
3. Администратор подтверждает
4. Система пересчитывает completeness для затронутых проектов

**Результат:** изменение применено, существующие проекты пересчитаны

---

## 5. Ограничения

- Стандарты не могут быть изменены без версионирования
- Изменение mapping должно сохранять историю (audit_log)
- Merge не должен автоматически скрывать различия
- Shared элементы не заменяют требования стандартов
- Все правила должны быть воспроизводимы и объяснимы
- Удаление стандарта невозможно при наличии привязанных данных (только деактивация)

---

## 6. Критерии приемки

Система считается принятой, если:

- [ ] Можно загрузить минимум 2 стандарта (GRI + IFRS S2)
- [ ] Можно декомпозировать их на requirement items
- [ ] Можно создать shared элементы и связать их с требованиями
- [ ] Можно настроить merge и увидеть пересечения
- [ ] При добавлении второго стандарта появляются только дополнительные требования (дельты)
- [ ] Можно задать правила полноты и они применяются
- [ ] Можно сформировать форму ввода на основе настроек
- [ ] Изменения администратора не ломают существующие проекты
- [ ] Все действия логируются в audit_log
