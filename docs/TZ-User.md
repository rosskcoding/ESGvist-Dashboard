# ТЗ: Пользователь системы

**Модуль:** ввод и управление ESG-данными
**Версия:** 1.0
**Статус:** Согласован

---

## 1. Цель

Обеспечить пользователю возможность:

- вводить данные для ESG-отчетности;
- переиспользовать ранее введенные значения;
- заполнять дополнительные требования при добавлении стандартов;
- понимать требования и статус заполнения;
- обеспечивать полноту раскрытия.

---

## 2. Область применения

Функциональность применяется для:

- сбора данных внутри организации;
- подготовки отчетности по нескольким стандартам;
- взаимодействия между сборщиками и ревьюерами;
- обеспечения качества и полноты данных.

---

## 3. Функциональные требования

### 3.1. Работа с проектом

Пользователь должен иметь возможность:

- открывать доступный проект отчётности;
- видеть:
  - период (Q2 2025, FY 2025);
  - стандарты (GRI, IFRS S2);
  - общий статус и прогресс (%);
  - дедлайн;
- выбирать область работы (по назначенным метрикам).

**Связь с БД:** `reporting_projects`, `reporting_project_standards`, `metric_assignments`

**UI:** Overview dashboard

### 3.2. Просмотр требований

Система должна отображать:

- список required элементов (только назначенные пользователю);
- их обязательность;
- принадлежность к стандартам (какой стандарт требует);
- статус заполнения (`missing` / `partial` / `complete`);
- описание и подсказки из requirement_items;
- индикацию reuse (если элемент используется в нескольких стандартах).

**UI:** Collection table — status-first layout

### 3.3. Ввод данных

Пользователь должен иметь возможность:

- вводить значения разных типов (number, text, date, enum, json);
- задавать dimensions (scope, gas, category);
- выбирать единицы измерения из справочника;
- указывать источник данных (source_record);
- указывать методологию и boundary;
- **сохранять данные как черновик** (`status = draft`);
- **отправлять данные на проверку** (`status = submitted`);
- **редактировать данные** до момента утверждения;
- **видеть**, заблокировано ли редактирование (после `approved`).

**Правила редактирования:**

| Статус | Редактирование | Действия |
|--------|---------------|----------|
| `draft` | ✅ разрешено | Сохранить / Отправить на ревью |
| `submitted` | ❌ заблокировано | Ожидание ревью |
| `in_review` | ❌ заблокировано | Ревьюер работает |
| `approved` | ❌ заблокировано | Только ESG-менеджер может откатить |
| `rejected` / `needs_revision` | ✅ разрешено | Исправить / Переотправить |

**Связь с БД:** `data_points`, `data_point_dimensions`, `methodologies`, `boundaries`

**UI:** Data Entry wizard (аналог QN-1→QN-4)

### 3.4. Переиспользование данных (reuse)

Система должна:

- при вводе данных проверять: существует ли DataPoint с тем же `shared_element_id`, `period`, `organization`, `unit`, `boundary`, `methodology`;
- если да — показывать существующее значение:
  > "Это значение уже введено для GRI 305-1. Использовать для IFRS S2.29?"
- объяснять совпадение (какие параметры совпали);
- позволять:
  - **использовать существующее** (создаётся binding в `requirement_item_data_points`);
  - **создать новое** (если параметры отличаются).

**Связь с БД:** `requirement_item_data_points` (binding_type = `direct`)

**Прозрачность переиспользования (Reuse Transparency):**

Система **обязана** показывать пользователю, где именно используется каждое введённое значение. Для каждого DataPoint, привязанного к нескольким requirement_items, отображается список всех disclosures и стандартов:

> «Это значение используется в: GRI 305-1, IFRS S2.29, SASB EM-EP-110a.1»

**Правила отображения:**
- Список стандартов/disclosures показывается **рядом с полем ввода** (inline или tooltip)
- При наведении — полный список с кодами disclosure requirements
- Если DataPoint используется в 1 стандарте — показывается как обычное поле
- Если в 2+ стандартах — визуальный индикатор reuse (иконка + badge с количеством)
- Пользователь должен понимать: «одно изменение повлияет на все перечисленные стандарты»

Это критично для доверия пользователя к системе: без прозрачности reuse пользователь не понимает последствия своих действий.

### 3.5. Работа с дополнительными требованиями (дельты)

При добавлении стандарта к проекту система должна:

- показывать новые поля, которые появились;
- объяснять их происхождение:
  > "IFRS S2 дополнительно требует: financial impact assessment"
- визуально отделять:
  - ✅ reused данные (уже заполнены);
  - 🆕 новые требования (нужен ввод);
  - +Δ дельты (доп. поля к существующим данным);
- подсвечивать недостающие элементы.

**Связь с БД:** `requirement_deltas`, `requirement_item_overrides`

### 3.6. Работа с narrative

Пользователь должен:

- вводить текстовые пояснения (methodology description, boundary statement);
- форматировать текст (basic rich text);
- добавлять комментарии к данным;
- отвечать на замечания ревьюера.

**Связь с БД:** `data_points` (text_value), `data_points.review_comment`

### 3.7. Работа с файлами (evidence)

Пользователь должен:

- загружать документы (PDF, Excel, изображения);
- привязывать файлы к data points или requirement items;
- видеть обязательность evidence (если requirement_item.item_type = `document`);
- просматривать загруженные файлы.

**Связь с БД:** `evidences`, `evidence_files`, `evidence_links`, `data_point_evidences`, `requirement_item_evidences` (см. TZ-Evidence.md)

**UI:** Evidence repository (`/evidence`) + привязка при вводе данных (drag-and-drop в wizard)

### 3.8. Контроль полноты

Система должна:

- показывать статус заполнения по каждому requirement_item;
- указывать недостающие элементы с конкретикой:
  > "Не заполнено: breakdown по газам (required by IFRS S2)"
- показывать ошибки валидации:
  > "⚠ Outlier: значение отличается >40% от прошлого периода"
- показывать покрытие по стандартам:
  > "GRI: 72% | IFRS S2: 45%"

**Связь с БД:** `requirement_item_statuses`, `disclosure_requirement_statuses`

**UI:** Fast scan summary над таблицами

### 3.9. Просмотр покрытия (coverage)

Пользователь (collector) видит:

- статус своих назначенных disclosure;
- степень покрытия по каждому;
- зависимость от конкретных данных;

ESG-менеджер и auditor дополнительно видят:

- **Merge View** — полную матрицу element × standard;
- общий coverage по всем стандартам;
- gap analysis.

### 3.10. Доработка и исправления

При отклонении данных (`status = rejected`) пользователь должен:

- получить уведомление (in-app + email);
- увидеть комментарий ревьюера (`review_comment`);
- отредактировать данные;
- повторно отправить на проверку (`rejected → submitted`);

**Связь с БД:** `data_points.status`, `data_points.review_comment`, `notifications`

### 3.11. Workflow и статусы

Система должна поддерживать workflow статусы для каждого data point:

**Поддерживаемые статусы:**

| Статус | Описание | Кто переводит |
|--------|----------|---------------|
| `draft` | Черновик, не отправлено | Сборщик (автоматически при создании) |
| `submitted` | Отправлено на проверку | Сборщик |
| `in_review` | Находится на проверке | Система (автоматически при наличии ревьюера) |
| `approved` | Подтверждено | Ревьюер |
| `rejected` | Отклонено | Ревьюер |
| `needs_revision` | Требует доработки | Ревьюер |

**Пользователь должен иметь возможность:**

- сохранять данные как `draft`;
- редактировать данные в статусе `draft` и `needs_revision`;
- отправлять данные в статус `submitted`;
- видеть текущий статус каждого data point;
- видеть историю изменений статусов (через `data_point_versions`);
- видеть комментарии ревьюера (`review_comment`);
- повторно отправлять данные после исправления (`needs_revision` → `submitted`).

**Связь с БД:** `data_points.status`, `data_point_versions`, `audit_log`

### 3.12. Доступ к Merge View

Merge View — матрица пересечения стандартов — доступна **не всем ролям**:

| Роль | Доступ к Merge View |
|------|-------------------|
| `collector` | ❌ Нет |
| `reviewer` | ✅ Да (read-only) |
| `esg_manager` | ✅ Да (полный) |
| `auditor` | ✅ Да (read-only) |
| `admin` | ✅ Да (полный) |

**Для пользователей роли `collector`:**

- Merge View не отображается в навигации;
- Пользователь видит только:
  - назначенные ему элементы;
  - пометку `reused` / `new` / `+Δ delta`;
  - источник требования (какой стандарт требует этот элемент).

**Для ролей с доступом к Merge View:**

- Отображается полная матрица element × standard;
- Видны reused элементы, дельты, пробелы;
- Покрытие по стандартам (coverage %);
- Drill-down до конкретного data point.

**Связь с БД:** `requirement_item_shared_elements`, `requirement_item_statuses`

**UI:** `/merge`

### 3.13. Блокировка и предупреждение о влиянии (Locking and Impact Warning)

Когда DataPoint используется в нескольких bindings (`requirement_item_data_points`), система должна защищать данные от непреднамеренных изменений.

**Предупреждение при редактировании:**

При попытке редактирования DataPoint, привязанного к 2+ requirement_items, система показывает:

> «Это значение используется в 3 стандартах. Изменение повлияет на все связанные disclosures.»

Далее показывается список затронутых disclosures:
- GRI 305-1 (status: approved)
- IFRS S2.29 (status: in_review)
- SASB EM-EP-110a.1 (status: partial)

Пользователь **обязан подтвердить** редактирование после просмотра impact list.

**Блокировка approved DataPoint:**

| Статус DataPoint | Редактирование | Кто может разблокировать |
|-----------------|---------------|------------------------|
| `draft` | Разрешено | — |
| `submitted` / `in_review` | Заблокировано | Автоматически при reject/needs_revision |
| `approved` | **Заблокировано** | Только ESG-менеджер (откат approved → draft) |

**Правила:**
- Если DataPoint в статусе `approved` — поля ввода **полностью заблокированы** (read-only)
- Для разблокировки ESG-менеджер должен откатить статус с обязательным обоснованием
- Откат записывается в `audit_log`
- После отката все связанные `requirement_item_statuses` пересчитываются через Completeness Engine
- UI показывает замок на approved DataPoint с tooltip: «Данные утверждены. Обратитесь к ESG-менеджеру для внесения изменений»

## 4. Пользовательские сценарии

### Сценарий 1. Ввод общего показателя

1. Пользователь открывает назначенную метрику (Scope 1 emissions)
2. Вводит значение, выбирает unit, methodology, boundary
3. Задаёт dimensions (если требуется)
4. Сохраняет черновик или отправляет на ревью
5. Система автоматически создаёт binding для всех стандартов, которые требуют этот shared element

**Результат:** один ввод закрывает требования GRI + IFRS одновременно

### Сценарий 2. Добавление стандарта к проекту

1. ESG-менеджер добавляет IFRS S2 к проекту (ранее был только GRI)
2. Система показывает пользователю:
   - "12 из 18 требований уже покрыты"
   - "3 дельты: нужно дозаполнить"
   - "3 новых требования: нужен ввод"
3. Пользователь заполняет только новые/дельта поля

**Результат:** дозаполнение без повторного ввода

### Сценарий 3. Переиспользование существующего значения

1. Пользователь начинает ввод для IFRS S2.29 (Scope 1)
2. Система показывает: "Значение 1240 t CO2e уже введено для GRI 305-1. Использовать?"
3. Пользователь подтверждает
4. Система создаёт binding (requirement_item_data_points)

**Результат:** reuse без дублирования

### Сценарий 4. Исправление после замечания

1. Ревьюер отклоняет данные с комментарием: "Значение CO2 выглядит заниженным, проверьте источник"
2. Пользователь получает уведомление
3. Открывает данные, видит комментарий
4. Исправляет значение, прикладывает новый документ
5. Повторно отправляет на ревью

**Результат:** данные пересмотрены и переотправлены

---

## 5. Ограничения

- Пользователь (collector) не может изменять структуру стандартов
- Нельзя удалить данные, используемые в нескольких стандартах, без подтверждения
- Нельзя завершить disclosure без обязательных элементов
- Доступ к данным ограничен ролью и назначениями (`metric_assignments`)
- Изменение значения DataPoint влияет на все связанные requirement_item_data_points
- Данные в статусе `submitted` / `in_review` / `approved` нельзя редактировать (только через reject → fix → resubmit)

---

## 6. Критерии приемки

Система считается принятой, если:

- [ ] Пользователь видит только назначенные ему метрики
- [ ] Одинаковые данные не требуют повторного ввода (reuse работает)
- [ ] При добавлении стандарта появляются только новые требования
- [ ] Система корректно показывает статус заполнения (missing/partial/complete)
- [ ] Пользователь может сохранить черновик и отправить на ревью
- [ ] Пользователь может загрузить файлы и привязать к данным
- [ ] Пользователь может исправить данные после замечаний ревьюера
- [ ] Система показывает, какие disclosure закрыты
- [ ] Пользователь понимает, какие данные используются повторно и где
- [ ] Все действия логируются в audit_log
