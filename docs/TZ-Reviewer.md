# ТЗ: Ревьюер / Аудитор

**Модуль:** проверка, комментарии и утверждение данных
**Версия:** 1.0
**Статус:** На согласовании

---

## 1. Цель

Обеспечить процесс проверки данных, включая:

- контроль корректности значений;
- проверку полноты;
- проверку соответствия стандартам;
- управление статусами утверждения.

---

## 2. Область применения

Используется для:

- внутреннего контроля качества ESG-данных;
- подготовки к внешнему аудиту;
- финального утверждения данных перед публикацией;
- обеспечения accountability (кто проверил, когда, почему).

---

## 3. Функциональные требования

### 3.1. Просмотр данных

Ревьюер должен видеть:

- все данные по назначенной области (через `metric_assignments.reviewer_id`);
- связанные disclosure requirements;
- связанные стандарты (какой стандарт требует этот элемент);
- текущий статус данных (`submitted`, `in_review`, `approved`, `rejected`);
- историю изменений (`data_point_versions`);
- предыдущие значения (year-over-year comparison);
- флаги валидации (outliers, inconsistencies).

**Связь с БД:** `data_points`, `data_point_versions`, `requirement_item_data_points`, `metric_assignments`

### 3.2. Review UI (split panel)

Система должна предоставлять интерфейс split panel:

**Левая панель (список):**
- Список data points, ожидающих ревью;
- Сортировка по:
  - urgency (overdue first);
  - дате отправки;
  - теме / стандарту;
- Визуальные маркеры:
  - ⚠ outlier;
  - 🔴 overdue;
  - время ожидания (дней в очереди);

**Правая панель (детали выбранного элемента):**
- Текущее значение;
- Единица измерения;
- Dimensions (scope, gas, category);
- Сравнение с предыдущим периодом;
- Требования стандарта (из requirement_items);
- Правила валидации;
- Комментарии (история);
- Attachments (evidence);
- Метаданные: кто ввёл, когда, источник.

**UI:** `/validation` — split panel (аналог email-клиента)

### 3.3. Действия ревьюера

Ревьюер должен иметь возможность:

| Действие | Результат | Обязательно |
|----------|-----------|-------------|
| **Approve** (утвердить) | `status → approved` | — |
| **Reject** (отклонить) | `status → rejected` | Обязателен комментарий |
| **Request revision** (вернуть на доработку) | `status → needs_revision` | Обязателен комментарий |
| **Comment** (оставить комментарий) | Статус не меняется | — |

**Правила:**

- `approve` — данные считаются финальными, включаются в completeness как `complete`
- `reject` — данные возвращаются сборщику, он получает уведомление
- `needs_revision` — аналог reject, но более мягкий (нужно уточнение, а не полная переделка)
- Ревьюер **не может** утверждать данные, если отсутствуют обязательные поля или evidence

**Связь с БД:** `data_points.status`, `data_points.reviewed_by`, `data_points.reviewed_at`, `data_points.review_comment`

### 3.4. Работа с комментариями

Система должна:

- хранить комментарии привязанные к:
  - data point (конкретное значение);
  - requirement item (требование в целом);
  - disclosure (группа требований);
- отображать историю комментариев в хронологическом порядке;
- показывать диалог reviewer ↔ collector;
- поддерживать **threaded comments** (ответ на конкретный комментарий);
- помечать комментарии по типу:
  - `question` — вопрос;
  - `issue` — проблема;
  - `suggestion` — рекомендация;
  - `resolution` — решение.

**Связь с БД:** отдельная таблица `comments` (или расширение `review_comment` до structured format)

```sql
create table comments (
    id                  bigserial primary key,
    data_point_id       bigint references data_points(id) on delete cascade,
    requirement_item_id bigint references requirement_items(id) on delete cascade,
    parent_comment_id   bigint references comments(id) on delete cascade,
    user_id             bigint not null references users(id) on delete cascade,
    comment_type        text not null check (comment_type in ('question', 'issue', 'suggestion', 'resolution', 'general')),
    body                text not null,
    is_resolved         boolean not null default false,
    created_at          timestamptz not null default now()
);
```

### 3.5. Массовые операции

Ревьюер должен иметь возможность:

- **утверждать несколько** элементов одновременно ("Approve all ready");
- фильтровать данные:
  - по статусу (`submitted`, `in_review`);
  - по стандарту;
  - по теме / concept_domain;
  - по сборщику;
  - по дате отправки;
  - по наличию outlier/issue;
- работать с **batch review**:
  - выбрать N элементов;
  - apply action (approve / reject с единым комментарием);
- видеть **summary перед batch action**:
  > "Вы собираетесь утвердить 5 элементов. 2 из них имеют outlier-флаги. Продолжить?"

### 3.6. Валидационная помощь

Система должна помогать ревьюеру, показывая:

- **Outlier alerts:** значение отличается от предыдущего периода > threshold;
- **Consistency checks:** scope1 + scope2 ≠ total;
- **Completeness flags:** не все required dimensions заполнены;
- **Evidence status:** есть ли required documents;
- **Cross-reference:** то же значение используется в нескольких стандартах (reuse indicator);

Ревьюер может:
- подтвердить outlier как корректный (с обоснованием);
- запросить дополнительные данные;
- пометить как `not_applicable` (с обоснованием).

### 3.8. Review Consistency Rules

При проверке данных ревьюер должен видеть контекст переиспользования (reuse), чтобы принимать осознанные решения:

**Обязательная информация при ревью:**

| Индикатор | Описание | Пример отображения |
|-----------|----------|--------------------|
| Reuse count | В скольких disclosure используется этот DataPoint | «Этот DataPoint используется в 3 disclosures» |
| Impact of approval | Какие стандарты будут затронуты при approve | «Approve повлияет на: GRI 305-1, IFRS S2.29, SASB EM-EP-110a.1» |
| Cross-reference | Ссылки на связанные требования из других стандартов | Кликабельные ссылки на related requirement_items |

**Правила:**
- Если DataPoint используется в нескольких стандартах (reuse), ревьюер **обязан** видеть полный список затронутых disclosures
- При approve DataPoint автоматически обновляет `requirement_item_statuses` для **всех** связанных requirement_items
- Ревьюер не может approve DataPoint только для одного стандарта — approve глобален
- Cross-reference индикатор помогает проверить консистентность данных между стандартами

### 3.8.1. Правила обоснования при массовых операциях (Bulk Reject/Approve)

Дополнительные правила для batch-операций (см. также 3.5):

| Операция | Комментарий | Правило |
|----------|------------|---------|
| Batch reject | **ОБЯЗАТЕЛЕН** | Каждый reject должен иметь обоснование. При batch — единый комментарий применяется ко всем элементам |
| Batch approve | Опционален, но **рекомендован** | Система показывает рекомендацию оставить комментарий |

**Summary перед batch action (обязательно):**

Перед выполнением массовой операции система показывает:
- Общее количество затронутых элементов
- Количество элементов с outlier-флагами
- Количество элементов с warnings
- Количество стандартов, затронутых действием

> Пример: «5 элементов будут утверждены, 2 из них имеют outlier-флаги. Действие затронет 3 стандарта. Продолжить?»

Ревьюер **обязан** подтвердить действие после просмотра summary.

### 3.7. Аудиторский режим (для роли auditor)

Аудитор имеет **read-only** доступ:

- просмотр всех данных;
- просмотр истории изменений;
- просмотр комментариев;
- просмотр audit_log;
- просмотр merge view;
- **не может** approve / reject / comment;
- может **экспортировать** данные для внешнего аудита.

## 4. Пользовательские сценарии

### Сценарий 1. Проверка значения

1. Ревьюер открывает `/validation`
2. Видит список data points в статусе `submitted`
3. Выбирает элемент "Scope 1 emissions — GRI 305-1"
4. Правая панель показывает: значение 1240 t CO2e, prev year 1380 (-10.1%), evidence (3 файла), комментарий сборщика
5. Ревьюер проверяет данные, нажимает **Approve**
6. `status → approved`, сборщик получает уведомление

### Сценарий 2. Возврат на доработку

1. Ревьюер видит outlier-флаг на "Energy consumption"
2. Открывает детали: значение отличается на 45% от прошлого периода
3. Пишет комментарий: "Значение выросло на 45%. Пожалуйста, проверьте источник и приложите обоснование."
4. Нажимает **Request revision**
5. `status → needs_revision`, сборщик получает уведомление с комментарием

### Сценарий 3. Массовое утверждение

1. Ревьюер фильтрует: "Environmental + no outliers + submitted"
2. Результат: 8 элементов
3. Выбирает все, нажимает "Approve selected"
4. Система показывает summary: "8 элементов будут утверждены"
5. Ревьюер подтверждает
6. Все 8 → `approved`

### Сценарий 4. Аудиторская проверка

1. Аудитор открывает Merge View
2. Видит coverage: GRI 85%, IFRS 60%
3. Фильтрует по status = `missing`
4. Drill-down в конкретный disclosure
5. Просматривает audit_log для этого disclosure
6. Экспортирует данные в Excel для внешнего отчёта

---

## 5. Ограничения

- Ревьюер **не редактирует** данные напрямую
- Ревьюер **не меняет** структуру требований
- Ревьюер **не может** утверждать данные без обязательных полей
- Reject и needs_revision **требуют** комментарий
- Ревьюер видит только данные по своим назначениям (через `metric_assignments`)
- Аудитор — read-only, не может менять статусы

---

## 6. Критерии приемки

- [ ] Ревьюер видит все назначенные ему data points в split panel
- [ ] Можно approve / reject / request revision
- [ ] При reject обязателен комментарий
- [ ] Сборщик получает уведомление при reject / needs_revision
- [ ] Работает batch approve (несколько элементов)
- [ ] Outlier-флаги отображаются корректно
- [ ] История комментариев сохраняется
- [ ] Threaded comments работают
- [ ] Статус корректно обновляется после действий ревьюера
- [ ] Аудитор имеет read-only доступ без возможности менять данные
- [ ] Все действия ревьюера логируются в audit_log
